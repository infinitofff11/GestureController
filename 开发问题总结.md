# GestureController 开发问题总结

> 本文档总结了在开发手势控制器App过程中，人工开发可能遇到的各类问题及解决方法。

---

## 一、MediaPipe HandLandmarker 集成问题

### 1.1 模型文件路径错误

**问题描述**：将 `hand_landmarker.task` 放入 `assets/models/` 子目录后，使用 `setModelAssetPath("hand_landmarker.task")` 加载失败，抛出 `FileNotFoundException`。

**解决方法**：`setModelAssetPath()` 的路径是相对于 `assets/` 目录的。如果文件在 `assets/models/` 下，路径应写为 `"models/hand_landmarker.task"`。

```kotlin
// 错误写法
.setModelAssetPath("hand_landmarker.task")

// 正确写法
.setModelAssetPath("models/hand_landmarker.task")
```

### 1.2 HandLandmarkerOptions 是 HandLandmarker 的嵌套类

**问题描述**：直接使用 `HandLandmarkerOptions.builder()` 会报 `Unresolved reference 'HandLandmarkerOptions'`，因为该类不是独立的顶层类。

**解决方法**：`HandLandmarkerOptions` 是 `HandLandmarker` 的嵌套类，必须使用 `HandLandmarker.HandLandmarkerOptions.builder()`：

```kotlin
// 错误写法
import com.google.mediapipe.tasks.vision.handlandmarker.HandLandmarkerOptions
val options = HandLandmarkerOptions.builder()...

// 正确写法
val options = HandLandmarker.HandLandmarkerOptions.builder()
    .setBaseOptions(baseOptions)
    .setRunningMode(RunningMode.LIVE_STREAM)
    .setResultListener { result, _ -> ... }
    .setErrorListener { e -> ... }
    .build()
```

### 1.3 LIVE_STREAM 模式必须设置 ResultListener 和 ErrorListener

**问题描述**：使用 `RunningMode.LIVE_STREAM` 时未设置 `setResultListener` 和 `setErrorListener`，导致运行时崩溃。

**解决方法**：LIVE_STREAM 模式是异步回调模式，必须设置结果监听器：

```kotlin
val options = HandLandmarkerOptions.builder()
    .setRunningMode(RunningMode.LIVE_STREAM)
    .setResultListener { result, _ -> /* 处理结果 */ }
    .setErrorListener { e -> /* 处理错误 */ }
    .build()
```

### 1.4 LIVE_STREAM 模式的时间戳要求

**问题描述**：调用 `detectAsync()` 时传入的时间戳不单调递增，导致 MediaPipe 抛出异常或丢弃帧。使用 `imageProxy.imageInfo.timestampMillis` 会报 `Unresolved reference 'timestampMillis'`。

**解决方法**：必须使用 `SystemClock.uptimeMillis()` 作为时间戳，这是 MediaPipe 官方推荐的方式。不要使用 `imageProxy.imageInfo.timestamp`（纳秒级），也不要使用 `System.currentTimeMillis()`：

```kotlin
// 错误写法
val frameTimestamp = imageProxy.imageInfo.timestampMillis  // 不存在此属性
val frameTimestamp = System.currentTimeMillis()  // 不保证单调递增

// 正确写法
val frameTime = SystemClock.uptimeMillis()
handLandmarker?.detectAsync(mpImage, frameTime)
```

### 1.5 detectLiveStream 方法不存在，应使用 detectAsync

**问题描述**：调用 `handLandmarker.detectLiveStream()` 会报 `Unresolved reference 'detectLiveStream'`。

**解决方法**：MediaPipe Android API 中，LIVE_STREAM 模式的检测方法是 `detectAsync()`，不是 `detectLiveStream()`：

```kotlin
// 错误写法
handLandmarker?.detectLiveStream(mpImage, frameTimestamp)

// 正确写法
handLandmarker?.detectAsync(mpImage, frameTime)
```

### 1.6 ImageProxy 转 Bitmap 的正确方式

**问题描述**：直接调用 `imageProxy.toBitmap()` 在某些设备上会抛出异常或返回错误的颜色，因为 CameraX 默认输出 YUV_420_888 格式。

**解决方法**：设置 `OUTPUT_IMAGE_FORMAT_RGBA_8888` 后，使用 `copyPixelsFromBuffer` 从 ImageProxy 复制像素到 Bitmap，并手动处理旋转和前置摄像头镜像：

```kotlin
// 正确方式：手动复制像素
val bitmapBuffer = Bitmap.createBitmap(
    imageProxy.width, imageProxy.height, Bitmap.Config.ARGB_8888
)
imageProxy.use { bitmapBuffer.copyPixelsFromBuffer(imageProxy.planes[0].buffer) }
imageProxy.close()

// 旋转和镜像
val matrix = Matrix().apply {
    postRotate(imageProxy.imageInfo.rotationDegrees.toFloat())
    postScale(-1f, 1f, imageProxy.width.toFloat(), imageProxy.height.toFloat())
}
val rotatedBitmap = Bitmap.createBitmap(
    bitmapBuffer, 0, 0, bitmapBuffer.width, bitmapBuffer.height, matrix, true
)
val mpImage = BitmapImageBuilder(rotatedBitmap).build()
```

### 1.7 模型加载耗时导致ANR

**问题描述**：在主线程加载 HandLandmarker 模型（约10MB），首次加载需1~2秒，可能导致ANR。

**解决方法**：在 ViewModel 中延迟初始化模型，或使用后台协程加载。本项目中在 ViewModel 的 `initLandmarker()` 方法中初始化，该方法在权限获取后的 `LaunchedEffect` 中调用，不会阻塞主线程。但模型创建本身仍在主线程，如需进一步优化可使用 `Dispatchers.IO`：

```kotlin
viewModelScope.launch(Dispatchers.IO) {
    landmarkerHelper = HandLandmarkerHelper(context, listener)
    withContext(Dispatchers.Main) {
        _gestureState.value = GestureUiState.Idle
    }
}
```

### 1.5 MediaPipe tasks-vision 版本兼容性

**问题描述**：`tasks-vision:0.10.8` 与某些版本的 Android Gradle Plugin 或 Kotlin 存在兼容性问题，编译时报 `Duplicate class` 错误。

**解决方法**：在 `build.gradle.kts` 中添加排除策略：

```kotlin
implementation("com.google.mediapipe:tasks-vision:0.10.8") {
    exclude(group = "com.google.protobuf", module = "protobuf-java")
}
```

## 二、CameraX 摄像头问题

### 2.1 Preview.surfaceProvider 属性语法不可用

**问题描述**：在 Kotlin 中使用 `preview.surfaceProvider = previewView.surfaceProvider` 会报 `Unresolved reference 'surfaceProvider'`，因为 `Preview` 类只有 `setSurfaceProvider()` 方法而没有对应的 getter，Kotlin 不会将其映射为属性。

**解决方法**：使用方法调用 `setSurfaceProvider()` 代替属性赋值：

```kotlin
// 错误写法
val preview = Preview.Builder().build().also { it.surfaceProvider = previewView.surfaceProvider }

// 正确写法
val preview = Preview.Builder().build()
preview.setSurfaceProvider(previewView.surfaceProvider)
```

### 2.2 ImageProxy.toBitmap() 图像格式问题

**问题描述**：CameraX 默认的 `ImageAnalysis` 输出格式为 YUV_420_888，直接调用 `toBitmap()` 在某些设备上会抛出异常或返回错误的颜色。

**解决方法**：设置 ImageAnalysis 的输出格式为 RGBA_8888：

```kotlin
val imageAnalyzer = ImageAnalysis.Builder()
    .setOutputImageFormat(ImageAnalysis.OUTPUT_IMAGE_FORMAT_RGBA_8888)
    .build()
```

### 2.2 ImageProxy 未关闭导致帧积压

**问题描述**：在 `onFrameAvailable` 回调中忘记调用 `imageProxy.close()`，导致 CameraX 停止产生新帧，画面卡死。

**解决方法**：确保在所有代码路径（包括异常路径）中都关闭 ImageProxy：

```kotlin
viewModelScope.launch(Dispatchers.Default) {
    try {
        landmarkerHelper?.detectLiveStream(imageProxy)
    } catch (e: Exception) {
        Log.e(TAG, "Detection error", e)
    } finally {
        imageProxy.close()  // 必须关闭
    }
}
```

### 2.3 前置摄像头镜像问题

**问题描述**：前置摄像头预览画面是镜像的，导致左右滑动手势识别方向与用户直觉相反。

**解决方法**：在 GestureHelper 中，由于前置摄像头图像已经镜像，X坐标的左右方向与实际相反。需要在手势判定时反转左右方向，或在 CameraX Preview 中设置镜像：

```kotlin
// 方案1：在PreviewView中设置镜像（仅影响预览，不影响分析）
previewView.implementationMode = PreviewView.ImplementationMode.COMPATIBLE

// 方案2：在手势判定中反转左右（推荐）
// 前置摄像头中，用户向左滑 → 图像中X增大 → SWIPE_RIGHT
// 需要交换 SWIPE_LEFT 和 SWIPE_RIGHT 的判定
```

### 2.4 CameraX 生命周期绑定问题

**问题描述**：在 Compose 中使用 CameraX 时，`ProcessCameraProvider.bindToLifecycle()` 需要 `LifecycleOwner`，但 Compose 重组可能导致重复绑定。

**解决方法**：在 `AndroidView` 的 `factory` 中创建 CameraManager 并绑定生命周期，factory 只会在首次组合时调用，不会因重组重复创建：

```kotlin
AndroidView(
    factory = { ctx ->
        PreviewView(ctx).also { previewView ->
            val cameraManager = CameraManager(activity, previewView) { ... }
            cameraManager.startCamera(ctx)
        }
    }
)
```

### 2.5 分辨率设置不生效

**问题描述**：`setTargetResolution(Size(640, 480))` 在某些设备上不生效，实际分辨率可能不同。

**解决方法**：`setTargetResolution` 是建议值，CameraX 会选择最接近的可用分辨率。这是正常行为，不需要修复。如果需要精确控制，可以使用 `setResolutionSelector()` (CameraX 1.3+)。

## 三、AccessibilityService 无障碍服务问题

### 3.1 服务未在系统设置中启用

**问题描述**：App安装后无障碍服务默认未启用，`GestureAccessibilityService.instance` 为 null，手势无法执行。

**解决方法**：在主界面提供引导按钮，跳转到系统无障碍设置页面：

```kotlin
val intent = Intent(Settings.ACTION_ACCESSIBILITY_SETTINGS)
startActivity(intent)
```

同时在代码中定期检查服务是否已启用，并在UI上显示状态。

### 3.2 检查无障碍服务是否启用的方法

**问题描述**：使用 `AccessibilityManager.isEnabled` 只能检查系统是否启用了无障碍功能，不能检查特定服务是否启用。

**解决方法**：使用 `getEnabledAccessibilityServiceList()` 检查特定包名的服务：

```kotlin
fun checkAccessibilityEnabled(context: Context): Boolean {
    val am = context.getSystemService(Context.ACCESSIBILITY_SERVICE) as AccessibilityManager
    val enabledServices = am.getEnabledAccessibilityServiceList(
        AccessibilityServiceInfo.FEEDBACK_ALL_MASK
    )
    return enabledServices.any {
        it.resolveInfo.serviceInfo.packageName == context.packageName
    }
}
```

### 3.3 dispatchGesture 在 Android 7.0 以下不可用

**问题描述**：`GestureDescription` 和 `dispatchGesture()` API 要求最低 API 24 (Android 7.0)，在更低版本设备上不可用。

**解决方法**：本项目的 `minSdk = 24`，因此不存在此问题。如果需要支持更低版本，只能使用 `performGlobalAction()` 实现返回等操作，无法模拟滑动和点击。

### 3.4 全局操作在某些ROM上不可用

**问题描述**：`GLOBAL_ACTION_RECENTS`（最近任务键）在某些国产ROM（如MIUI、EMUI）上不生效。

**解决方法**：这是系统限制，没有通用解决方案。可以尝试使用 `GLOBAL_ACTION_BACK` 作为替代，或引导用户在系统设置中关闭手势导航。

### 3.5 AccessibilityService 被系统杀死

**问题描述**：长时间运行后，系统可能为节省内存而杀死无障碍服务。

**解决方法**：
- 在 `accessibility_service_config.xml` 中设置 `android:notificationTimeout="100"`
- 在服务的 `onServiceConnected()` 中发送前台通知（Android 8.0+），提高服务优先级
- 引导用户在系统设置中将App加入电池优化白名单

### 3.6 canPerformGestures 必须设为 true

**问题描述**：`accessibility_service_config.xml` 中未设置 `android:canPerformGestures="true"`，导致 `dispatchGesture()` 调用失败。

**解决方法**：确保配置文件中包含此属性：

```xml
<accessibility-service
    android:canPerformGestures="true"
    ... />
```

## 四、悬浮窗问题

### 4.1 SYSTEM_ALERT_WINDOW 权限获取

**问题描述**：Android 6.0+ 需要用户手动授予悬浮窗权限，`TYPE_APPLICATION_OVERLAY` 需要此权限。

**解决方法**：引导用户到系统设置页面授权：

```kotlin
if (!Settings.canDrawOverlays(context)) {
    val intent = Intent(
        Settings.ACTION_MANAGE_OVERLAY_PERMISSION,
        Uri.parse("package:${context.packageName}")
    )
    startActivity(intent)
}
```

### 4.2 悬浮窗类型选择

**问题描述**：使用 `TYPE_SYSTEM_ALERT`（已废弃）或 `TYPE_PHONE`（已废弃）在某些 Android 版本上不显示。

**解决方法**：Android 8.0+ 必须使用 `TYPE_APPLICATION_OVERLAY`：

```kotlin
val params = WindowManager.LayoutParams(
    WindowManager.LayoutParams.WRAP_CONTENT,
    WindowManager.LayoutParams.WRAP_CONTENT,
    WindowManager.LayoutParams.TYPE_APPLICATION_OVERLAY,  // 必须使用此类型
    WindowManager.LayoutParams.FLAG_NOT_FOCUSABLE,
    PixelFormat.TRANSLUCENT
)
```

### 4.3 悬浮窗触摸事件与拖动冲突

**问题描述**：悬浮窗设置了 `FLAG_NOT_FOCUSABLE`，但内部按钮的点击事件和整体拖动事件冲突。

**解决方法**：在悬浮窗根布局上设置触摸监听实现拖动，但只在非按钮区域响应拖动。或者使用 `FLAG_NOT_TOUCH_MODAL` 替代 `FLAG_NOT_FOCUSABLE`，允许悬浮窗接收触摸事件。

### 4.4 悬浮窗在锁屏状态下不显示

**问题描述**：设备锁屏后悬浮窗消失。

**解决方法**：添加 `FLAG_SHOW_WHEN_LOCKED` 和 `FLAG_KEEP_SCREEN_ON` 标志（API 27以下），或在 WindowManager.LayoutParams 中设置合适的 flag。注意 Android 8.0+ 对锁屏悬浮窗有更严格的限制。

## 五、手势识别算法问题

### 5.1 握拳检测误判

**问题描述**：用户正常握持手机时，手指可能自然弯曲，被误判为握拳。

**解决方法**：
- 调大 `FIST_THRESHOLD` 阈值（当前0.12），使判定更严格
- 增加拇指的判定条件（拇指指尖到小指MCP的距离）
- 增加握拳持续时间的判定，要求连续多帧检测到握拳才触发

```kotlin
// 改进：增加拇指判定
private fun isFist(landmarks: List<NormalizedLandmark>): Boolean {
    val palm = landmarks[0]
    val fingerTips = listOf(8, 12, 16, 20)
    val allFingersClosed = fingerTips.all { tip ->
        val dx = landmarks[tip].x() - palm.x()
        val dy = landmarks[tip].y() - palm.y()
        sqrt(dx * dx + dy * dy) < FIST_THRESHOLD
    }
    // 额外判定：拇指指尖到小指MCP的距离
    val thumbTip = landmarks[4]
    val pinkyMcp = landmarks[17]
    val thumbDist = sqrt(
        (thumbTip.x() - pinkyMcp.x()).pow(2) +
        (thumbTip.y() - pinkyMcp.y()).pow(2)
    )
    return allFingersClosed && thumbDist < 0.1f
}
```

### 5.2 点击与静止误判

**问题描述**：手指静止不动时，微小的抖动可能被判定为点击手势。

**解决方法**：
- 降低 `TAP_DISTANCE_THRESHOLD`（当前0.05），使判定更严格
- 增加点击前需要有快速移动的判定条件
- 在防抖器中增加更长的冷却时间

### 5.3 滑动方向判定不稳定

**问题描述**：斜向滑动时，X和Y方向的位移接近，导致左右和上下判定频繁切换。

**解决方法**：在 `detectSwipe()` 中增加方向锁定的逻辑，一旦确定主方向就不再切换：

```kotlin
private var lockedDirection: GestureType? = null

private fun detectSwipe(): GestureType {
    val first = fingerTipHistory.first()
    val last = fingerTipHistory.last()
    val dx = last.first - first.first
    val dy = last.second - first.second

    // 方向锁定：如果已确定方向，不再切换
    if (lockedDirection != null) return lockedDirection!!

    val result = when {
        abs(dx) > SWIPE_THRESHOLD && abs(dx) > abs(dy) * 1.5f ->
            if (dx > 0) GestureType.SWIPE_RIGHT else GestureType.SWIPE_LEFT
        abs(dy) > SWIPE_THRESHOLD && abs(dy) > abs(dx) * 1.5f ->
            if (dy > 0) GestureType.SWIPE_DOWN else GestureType.SWIPE_UP
        else -> GestureType.NONE
    }

    if (result != GestureType.NONE) lockedDirection = result
    return result
}
```

### 5.4 防抖器冷却时间过长导致手势响应慢

**问题描述**：`cooldownMs = 400L` 导致连续快速手势被过滤，用户感觉响应迟钝。

**解决方法**：根据使用场景调整冷却时间：
- 滑动操作：300ms
- 点击操作：200ms
- 返回操作：500ms（避免误触）

可以为不同手势类型设置不同的冷却时间。

### 5.5 前置摄像头坐标镜像

**问题描述**：前置摄像头的图像是镜像的，导致左右滑动手势识别方向与用户实际手势方向相反。

**解决方法**：在 `GestureHelper` 中反转X坐标，或在判定逻辑中交换左右：

```kotlin
// 方案1：在添加历史记录时反转X坐标
fingerTipHistory.add(Pair(1f - indexTip.x(), indexTip.y()))

// 方案2：在判定时交换左右方向
if (dx > 0) GestureType.SWIPE_LEFT else GestureType.SWIPE_RIGHT
```

## 六、Compose 与传统 View 混合使用问题

### 6.1 PreviewView 在 Compose 中的使用

**问题描述**：CameraX 的 `PreviewView` 是传统 View，需要在 Compose 中通过 `AndroidView` 嵌入，但生命周期管理复杂。

**解决方法**：使用 `AndroidView` 的 `factory` 创建 PreviewView，并在其中初始化 CameraManager。注意 factory 只调用一次，不会因重组重复创建。

### 6.2 Compose 重组导致状态丢失

**问题描述**：Compose 重组时，`remember` 中的 CameraManager 实例可能被重新创建。

**解决方法**：将 CameraManager 的创建放在 `AndroidView.factory` 中，或使用 `rememberUpdatedState` 保持最新引用。本项目中 CameraManager 在 factory 中创建，不会因重组丢失。

### 6.3 LiveData 在 Compose 中的使用

**问题描述**：使用 `observeAsState()` 观察 LiveData 时，初始值可能为 null。

**解决方法**：提供默认初始值：

```kotlin
val gestureState by viewModel.gestureState.observeAsState(GestureUiState.Idle)
```

## 七、权限管理问题

### 7.1 运行时权限请求

**问题描述**：Android 6.0+ 需要运行时请求 CAMERA 权限，仅声明 `<uses-permission>` 不够。

**解决方法**：使用 `ActivityResultContracts.RequestPermission()` 请求权限：

```kotlin
val launcher = rememberLauncherForActivityResult(
    ActivityResultContracts.RequestPermission()
) { granted -> /* 处理结果 */ }

launcher.launch(Manifest.permission.CAMERA)
```

### 7.2 权限被永久拒绝

**问题描述**：用户选择"不再询问"并拒绝权限后，`requestPermission()` 不再弹出对话框。

**解决方法**：检测 `shouldShowRequestPermissionRationale()` 返回 false 且权限未授予时，引导用户到应用设置页面手动授权：

```kotlin
if (!shouldShowRequestPermissionRationale(Manifest.permission.CAMERA)
    && ContextCompat.checkSelfPermission(this, Manifest.permission.CAMERA) != PackageManager.PERMISSION_GRANTED) {
    // 引导到应用设置
    val intent = Intent(Settings.ACTION_APPLICATION_DETAILS_SETTINGS,
        Uri.parse("package:$packageName"))
    startActivity(intent)
}
```

### 7.3 悬浮窗权限回调

**问题描述**：`Settings.canDrawOverlays()` 的状态变化没有系统回调，从设置页面返回后需要手动刷新。

**解决方法**：使用 `LaunchedEffect` 定期轮询检查权限状态（本项目中每秒检查一次）。

## 八、性能优化问题

### 8.1 帧处理延迟

**问题描述**：MediaPipe 推理耗时约25-50ms，如果处理速度跟不上帧率，会导致帧积压。

**解决方法**：
- 使用 `STRATEGY_KEEP_ONLY_LATEST` 丢弃过时帧
- 降低 ImageAnalysis 分辨率（640x480）
- 使用 `Dispatchers.Default` 在后台线程执行推理

### 8.2 Bitmap 内存泄漏

**问题描述**：每帧创建新的 Bitmap 对象，频繁 GC 导致卡顿。

**解决方法**：
- 及时调用 `imageProxy.close()` 释放缓冲区
- MediaPipe 内部会管理 Bitmap 的生命周期
- 如果手动创建 Bitmap，使用 `BitmapFactory.Options.inBitmap` 复用

### 8.3 ViewModel 中的协程泄漏

**问题描述**：在 ViewModel 中使用 `viewModelScope.launch` 但未正确取消，导致 Activity 销毁后仍在处理帧。

**解决方法**：`viewModelScope` 会自动在 `onCleared()` 时取消所有协程。确保不要使用 `GlobalScope`。

### 8.4 手势历史记录无限增长

**问题描述**：`fingerTipHistory` 列表未限制长度，长时间运行可能导致内存泄漏。

**解决方法**：限制列表长度为15帧（已在代码中实现）：

```kotlin
if (fingerTipHistory.size > 15) {
    fingerTipHistory.removeAt(0)
}
```

## 九、构建与编译问题

### 9.1 Compose 与 ViewBinding 冲突

**问题描述**：项目同时启用 Compose 和 ViewBinding，编译时报错。

**解决方法**：在 `build.gradle.kts` 中只启用 Compose，不需要 ViewBinding：

```kotlin
buildFeatures {
    compose = true
    viewBinding = false  // 不需要
}
```

### 9.2 Kotlin 版本与 Compose Compiler 不兼容

**问题描述**：Kotlin 2.0.x 需要对应版本的 Compose Compiler，版本不匹配会导致编译错误。

**解决方法**：使用 Kotlin Compose 插件自动匹配版本：

```kotlin
plugins {
    alias(libs.plugins.kotlin.compose)  // 自动匹配 Compose Compiler 版本
}
```

### 9.3 ProGuard 混淆导致 MediaPipe 崩溃

**问题描述**：Release 构建启用 ProGuard 后，MediaPipe 的反射调用被混淆，导致运行时崩溃。

**解决方法**：在 `proguard-rules.pro` 中添加 MediaPipe 保留规则：

```
-keep class com.google.mediapipe.** { *; }
-keep class com.example.gesturecontroller.model.** { *; }
-dontwarn com.google.mediapipe.**
```

### 9.4 aaptOptions 未配置导致模型文件未打包

**问题描述**：`assets/models/hand_landmarker.task` 文件未被包含在 APK 中。

**解决方法**：确保文件确实在 `src/main/assets/models/` 目录下，且 `build.gradle.kts` 中没有排除 assets 的配置。默认情况下，`assets/` 目录下的所有文件都会被打包。

## 十、调试与测试问题

### 10.1 模拟器无法测试摄像头手势

**问题描述**：Android 模拟器的前置摄像头质量差，无法准确识别手势。

**解决方法**：
- 使用真机测试
- 模拟器中可使用电脑摄像头（在 AVD 设置中配置）
- 编写单元测试覆盖 GestureHelper 和 GestureDebouncer 的逻辑

### 10.2 AccessibilityService 调试困难

**问题描述**：无障碍服务的断点调试不方便，服务可能在调试器附加前就已启动。

**解决方法**：
- 使用 `Log.d()` 输出日志
- 在 `onServiceConnected()` 中添加 `Debug.waitForDebugger()` 等待调试器
- 使用 `adb shell dumpsys accessibility` 检查服务状态

### 10.3 手势识别参数调优

**问题描述**：不同用户的手部大小、手势习惯不同，固定阈值无法适应所有人。

**解决方法**：
- 提供设置页面让用户调整灵敏度
- 实现自适应阈值算法
- 收集多用户数据，使用统计方法确定最优阈值

## 十一、常见运行时崩溃

### 11.1 `HandLandmarker.createFromOptions()` 返回 null

**原因**：模型文件路径错误或文件损坏。

**排查步骤**：
1. 检查 `assets/models/` 目录下是否存在 `hand_landmarker.task`
2. 检查文件大小是否正确（约10MB）
3. 检查 `setModelAssetPath()` 路径是否正确

### 11.2 `IllegalStateException: Image is already closed`

**原因**：`imageProxy.close()` 被调用了多次，或在 `detectLiveStream()` 使用 ImageProxy 之前就已关闭。

**解决方法**：确保 `close()` 只在 `finally` 块中调用一次，且在 `detectLiveStream()` 之后调用。

### 11.3 `SecurityException: Permission Denial`

**原因**：未获取 CAMERA 权限就启动摄像头。

**解决方法**：在启动摄像头前检查权限，未获取则请求权限。

### 11.4 `WindowManager$BadTokenException`

**原因**：未获取 `SYSTEM_ALERT_WINDOW` 权限就添加悬浮窗。

**解决方法**：在添加悬浮窗前检查 `Settings.canDrawOverlays()`。

## 十二、Android 版本兼容性问题

### 12.1 Android 13+ 通知权限

**问题描述**：Android 13 (API 33) 需要运行时请求 `POST_NOTIFICATIONS` 权限，否则前台服务通知不显示。

**解决方法**：如果无障碍服务使用前台通知，需要请求通知权限。

### 12.2 Android 12+ 前台服务限制

**问题描述**：Android 12+ 对前台服务启动有严格限制，从后台启动前台服务会抛出异常。

**解决方法**：无障碍服务是系统服务，不受此限制。但 OverlayService 如果需要前台服务，需确保从前景启动。

### 12.3 Android 10+ 后台启动 Activity 限制

**问题描述**：Android 10+ 限制从后台启动 Activity。

**解决方法**：本项目中不需要从后台启动 Activity，权限引导页面由用户主动触发。
